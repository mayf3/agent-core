# Acceptance Kit V2 治理绑定修复 — 终版计划

## 核心架构

```
用户请求
  ↓
External Acceptance Selector（理解产品名，选验收方案 — Kernel 不知道）
  ↓
Coding Harness（生成 Draft → 修复 → 冻结候选字节 → 计算 subject_digest）
  ↓
Verification Harness（编译 → 通用 Runtime Contract → 私有验收 → 出具 Receipt Envelope）
  │   { invocation_intent_id, issuer, subject_digest, outcome,
  │     evidence_digest, opaque_payload_digest, receipt_digest }
  ↓
Kernel（零 Kit 语义，纯机械对账）：
  - 这是我发出的 Invocation 吗？     → invocation_intent_id
  - 发行者是本次授权 Harness 吗？    → issuer（Invocation 绑定 + 通道认证）
  - subject 是当前制品吗？          → subject_digest
  - outcome 是 Passed 吗？
  - 回执是否完整且未重放？          → receipt_digest（内容完整证明，不是签名）
  ↓
用户 Approval
  ↓
Deployment Harness（只部署同 subject/artifact digest）
```

## 六、通用 Receipt Envelope（Kernel 理解的回执格式）

### 新建：`src/domain/external_receipt_envelope.rs`
```rust
pub struct ExternalReceiptEnvelope {
    pub invocation_intent_id: String,  // 防回执错配
    pub issuer: String,                // 发行者（Invocation 绑定 + 通道认证，非自签名）
    pub subject_digest: String,        // 被验收制品 digest
    pub outcome: ExternalOutcome,      // Passed / Failed
    pub evidence_digest: String,
    pub opaque_payload_digest: Option<String>,  // 验收包内部摘要（Kernel 不解）
    pub receipt_digest: String,        // SHA-256(以上) — 内容完整证明，不是签名
}
```

### 信任模型
- Kernel 通过 InvocationIntent 指定受信 Harness
- Harness 凭认证通道返回回执
- receipt_digest 防内容篡改；伪造防护靠 invocation binding + 认证 + 防重放

### `CodingTaskSubmitResult`
- 新增 `receipt_envelope: ExternalReceiptEnvelope`

## 十、Kernel 负面宪法 + PR Checklist + 边界检查

### 文档：`docs/architecture/KERNEL_NEGATIVE_CONSTITUTION.md`
```
Kernel 禁止新增：
  Planner / Plan / Role / Agent
  AcceptanceKit / Bundle / Spec / Verifier
  PublicSpec / PrivateVerifier / TestCase / Fixture
  Dashboard / ProductType / BusinessMetric

Kernel 允许：
  Principal / Scope / Run / Intent
  Decision / Invocation / ReceiptEnvelope
  ArtifactRef / SubjectDigest / ReceiptDigest
  outcome / evidence / issuer / failure_class

每个 Kernel PR 必答：
  1. 为什么不能只在外部 Harness 完成？
  2. 不进入 Kernel 会造成哪个不可绕过的安全缺口？
  3. 新增的是通用治理原语，还是当前产品的便利字段？
  4. 换一套完全不同的外部系统后仍成立吗？
  5. Kernel 是否开始理解新的业务事实？
```

### 脚本：`scripts/check-kernel-negative-boundary.sh`
- 检查 `src/` 不含禁用标识（带允许列表）

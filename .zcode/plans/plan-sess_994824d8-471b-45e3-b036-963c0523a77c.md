### Part 2: Acceptance Kit Module — Coding Harness (5 files)

#### 4. `tools/coding-harness/src/self_evolution/acceptance_kit/mod.rs` (NEW)
```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum AcceptanceKitId {
    TokenDashboardV0,
    FailureEventViewerV0,
}

impl AcceptanceKitId {
    pub fn kit_id(&self) -> &'static str;
    pub fn kit_version(&self) -> &'static str;
    pub fn target_profile(&self) -> &'static str;
    pub fn public_spec(&self) -> serde_json::Value;
    pub fn public_spec_digest(&self) -> String;       // sha256 of JSON
    pub fn combined_kit_digest(&self) -> String;       // sha256 of kit_id + public_spec_digest + private_verifier_digest
    
    // Verification: calls the appropriate private verifier
    pub fn verify(
        &self,
        request: &DevelopmentRequest,
        source: &str,
        stdout: &str,
    ) -> Result<(), String>;
    
    // Resolution: find a kit from a DevelopmentRequest
    // Uses acceptance_kit_ref first, then falls back to name-based exact match
    // NEVER uses substring matching on "token"
    pub fn resolve(request: &DevelopmentRequest) -> Result<Self, &'static str>;
}
```

- Private verifier digests are compile-time constants (the verifier is the validation logic in code, so its "code identity" is the combined kit digest)
- `resolve()`: If `request.acceptance_kit_ref` is set, match it exactly against known kit IDs → else `ACCEPTANCE_KIT_SELECTION_REQUIRED`. No fallback to substring matching.

## Key Design Decisions

1. **Kit selection is explicit**: `acceptance_kit_ref` must be set in the DevelopmentRequest. No substring matching on "token" anywhere. `AcceptanceKitId::resolve()` returns `Err("ACCEPTANCE_KIT_SELECTION_REQUIRED")` when no explicit ref is found.

2. **Kit identity = code identity**: The private verifier is Rust code. Its digest is the combined kit digest, derived from the kit_id + public_spec_digest. Changing either spec or verifier code changes the combined digest.

3. **Public spec is per-request**: Injected alongside DEVELOPMENT_REQUEST_JSON. Not cached in SYSTEM_PROMPT. Not retained between model calls.

4. **Structured diagnostics**: Text format with constraint_id, path, expected, actual rather than raw JSON — more readable for the model, no type system changes needed.

5. **Failure Event Viewer Kit**: Minimal public_spec describing failure event display. No token/metrics fields. Private verifier checks for failure event fields and absence of token-specific contamination.

## Risk Mitigation

- Existing tests for token validation are preserved (moved, not deleted)
- Cache identity check ensures kit digest changes invalidate old candidates
- `ACCEPTANCE_KIT_SELECTION_REQUIRED` is a clear error, not silent mis-selection
- No breaking changes to the external harness protocol

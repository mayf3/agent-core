### 2. `src/server/coding_task_submit.rs` (≤500 lines)
Orchestration glue — the `handle_coding_task_submit()` function:
1. Parse intent via router (or accept pre-parsed CodingIntent)
2. Create HCR via `journal.create_harness_change_request()`
3. Execute HCR worker via `execute_hcr()` (claim + run binding)
4. Create Run with RunMode::Hcr
5. Gateway approve for `external.coding_hcr_accept`
6. Create calculator candidate directory in harness workspace (copy from fixture or generate source files)
7. Call `hcr_acceptance::handle()` with candidate_ref
8. If settlement is `Succeeded` → create CapabilityChangeProposal + trusted HCR link
9. Return structured result with all IDs

Idempotency: same (principal_id, spec_digest) → same HCR, no duplicate proposals.

### 5. Tests file `src/server/r3a_phase_a_e2e.rs` (≤500 lines)
- **North Star Phase-A E2E:** Full chain authenticated message → router → submit → HCR → candidate → gates → settlement → proposal
- **Unsupported request:** Rejected before HCR creation
- **Missing control operation:** Test with snapshot without external.coding_task_submit
- **CandidateFailed:** Implementation error detected by gates → no proposal
- **InfrastructureFailure:** Harness unreachable → no settlement → no proposal
- **Forged settlement linkage:** Wrong digests → proposal = 0
- **Replay 20-way concurrency:** Same message → 1 HCR, 1 proposal, 1 link

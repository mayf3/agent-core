风险分级

Critical: core 膨胀。 影响：后续无法替换 workflow、多 agent、sandbox。建议：新增能力先证明不能作为插件。
High: secret 泄露。 影响：远程仓库暴露 API key/飞书 secret。建议：`.gitignore`、secret scan、redacted logs、PR 前检查。
High: sandbox 只有文档。 影响：shell 工具实现后风险很高。建议：M2 必须先有 workspace boundary、timeout、output cap、approval。
Medium: PR 流程未被远程强制。 影响：习惯可绕过。建议：远程创建后开启 branch protection 或至少只通过 PR 合并。

Linus 视角严苛审查

最大的危险是把“可扩展”理解成“到处加抽象”。 core 应该暴露少数稳定结构，而不是为未来所有可能性预留 hook。

  • 插件 API 只开放 register tool/provider/transport，别急着开放 lifecycle hook 大全。
  • 工具权限必须在 dispatch 层强制，不能相信 agent prompt。
  • 状态必须 append-only 优先，否则失败回放会变成猜谜。
  • Feishu 插件不要把外部消息格式漏进 core。
  • 沙盒 provider 是接口，Docker 编排不是 core 职责。

未执行独立 subagent 审查：当前 subagent 工具规则要求只有用户明确要求委派时才能调用。

修复优先级

  1. 初始化 git，完成 bootstrap commit；远程创建后切换到 PR-only 流程。
  2. 让 `pnpm check` 成为每个 PR 的必跑检查。
  3. 实现 M1：runs/events/envelope/state/doctor。
  4. 实现 M2 前先定义 sandbox boundary 和 approval policy。
  5. 实现 M4 飞书插件，让手机入口尽快可感知。
